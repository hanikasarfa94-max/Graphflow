--------------------------------------------------------------
  0. THE MACRO PRINCIPLE (READ THIS BEFORE ANY PIXEL DECISION)
--------------------------------------------------------------

The group is the subject. Projects are containers. A user belongs to many
groups; many groups nest inside many projects; production scale is up to
~100 groups per user. Therefore:

  • The MAIN VIEW is NOT a project page. It is the user's personal stream
    with their sub-agent. (Aligns with north-star "one input box, like
    Claude Code"; aligns with 04-28 memo "Home page dropped — land in the
    general-agent stream".)

  • The PRIMARY NAV is NOT a project list. It is a stream list — the
    groups + DMs the user is in. Projects are NOT a navigation rung; they
    are a soft scope hint surfaced via ProjectBar pills.

  • The DETAIL pages (项目管理 / 任务 / 知识库 / 组织 / 审计) are
    demoted to a far-left icon RAIL. They REPLACE the main column when
    selected; they are not peers of the stream nav.

  • The WORKBENCH (right side) is a user-composed shortcut tray for
    tasks / knowledge / requests / skills next to the active stream. It
    is opt-in and panels are shortcut-views into the underlying detail
    pages — never the only place a piece of state lives.

  │  ║ 整合全平台信息，跨项目咨询    ║    │   always present.
  │  ╚══════════════════════════════╝    │   project_id=NULL stream.
  │                                       │
  │  项目 Agent ▾                         │ ← header replaces "专门 Agent"
  │    · 星际漂流 — 第1季的 Agent         │   one entry per project the

  Backend mapping (StreamRow.kind, orm.py:1076-1106):

    我的 Agent   →  ONE 通用 (global) agent + N 项目 (per-project)
                    agents. Q-A revised 2026-05-01 → hybrid model.

                    Section structure (matches prototype's agentPrimary +
                    specialistAgents shape exactly, but substituting
                    project-scoped agents for the deleted specialists):

                      [agentPrimary big card]
                      通用 Agent
                      "整合全平台信息，跨项目咨询。"
                      [小 navItem entries below]
                      <Project A> 的 Agent
                      <Project B> 的 Agent
                      ... one per project the user is in

                    Backend mapping:

                      通用 Agent  →  StreamRow kind='personal',
                                     owner_user_id=me, project_id=NULL.
                                     ONE row per user. The schema already
                                     permits this (orm.py:1108 — column
                                     is nullable). No data migration.

                      项目 Agent  →  StreamRow kind='personal',
                                     owner_user_id=me, project_id=<X>.
                                     One row per (user, project) pair —
                                     this is the existing shipped
                                     behavior, untouched.

                    Why hybrid (user direction 2026-05-01):
                    "main agent + project-scoped agents make work more
                    clear." Cross-cutting / strategic talk goes in 通用;
                    project-bound talk goes in that project's agent.
                    User picks the venue, no inference needed.

                    Optional enhancement (defer to a separate plan):
                    when posting in 通用 Agent, the edge agent MAY
                    infer project context from evidence and surface a
                    soft suggestion ("this looks like Project A — open
                    in Project A's agent?"). NOT required for v1.

                    System signals (drift, gated, membrane, route
                    proposals) keep posting to the per-(user, project)
                    personal stream as today. The 通用 Agent stream
                    starts empty and grows from user-initiated turns.

    群组         →  StreamRow where kind IN ('project', 'room')
                    FLAT LIST. We do NOT visually distinguish "project
                    main room" from "sub-room". To the user, they are
                    all groups. Sort by stream.last_activity_at desc
                    (Q-D answer 2026-05-01 — recency only, no search
                    input as primary affordance). Section header
                    collapse remains available per the prototype's
                    NavSection pattern.

                    Display name resolution (Q-E answer 2026-05-01):
                      1. StreamRow.name if set
                      2. else ProjectRow.name of the project the
                         creator of the stream belongs to
                      3. else "未命名群组" or stream id suffix
                    Resolved server-side on the streams list endpoint.

                    Why flat: groups are the unit. The project boundary
                    is for scope/permission/quorum; it does not need to
                    be a visual hierarchy in the nav. The 04-28 memo's
                    "multiple team-rooms per project" decision combined
                    with the user's 05-01 directive ("don't check group
                    types into project or room") collapses the visual.

                    The ONLY place the project/room distinction surfaces:
                    – Vote scope (handled automatically by BE via
                       scope_stream_id, see §3 below).
                    – ProjectBar crumb (which project the active group
                       belongs to).

    私聊         →  StreamRow where kind='dm'
                    No project anchor. List by last_activity_at.

--------------------------------------------------------------
  3. VOTES UNDER GROUP-COLLAPSE — NO BACKEND CHANGE
--------------------------------------------------------------

The smallest-relevant-vote rule is already implemented in
DecisionVoteService._compute_tally (decision_votes.py:251-279):

    if scope_stream_id is not None:
        members = StreamMemberRepository.list_for_stream(scope_stream_id)
        scope_kind = "room"          # ← display label only
    else:
        members = ProjectMemberRepository.list_for_project(project_id)
        scope_kind = "project"       # ← display label only

The vote scope = members of the stream where the decision was crystallized.
"Group" in our UX maps cleanly:

    Group is project-main stream  →  scope_stream_id = <main_stream_id>
                                     quorum = all project members
    Group is sub-room             →  scope_stream_id = <room_stream_id>
                                     quorum = room members (subset)
    Decision floats project-wide  →  scope_stream_id = null
                                     quorum = all project members

The "room" / "project" string in the tally payload is a DISPLAY LABEL,
not a control flow distinction. We MAY keep both labels in the API tally
even though the UX shows them uniformly as "群组"; the FE's
DecisionCard scope-explainer line should read "Voting with this group's
N members" regardless of scope_kind value.

INVARIANT THE BE STILL RELIES ON:
  StreamMemberRepository.list_for_stream(project_main_stream_id)
    must equal
  ProjectMemberRepository.list_for_project(project_id)
This is enforced by the streams primitive auto-create + dedup contract
(GRAPH_REPORT.md HE 764, 782, 807). If this invariant is ever broken
by future code, votes will silently diverge from project membership.
Worth a regression test if not already present.

  4. THE RAIL — DEMOTED DETAIL-PAGE NAV
--------------------------------------------------------------

Far-left vertical icon strip (50px). Switches activeView between
agentView (chat) and module views. Module views REPLACE the main
column (and hide ImNav, per legacy-standalone-v6.html lines 95-97).

  Icons — exact glyphs from prototype data.ts:3-10. Six items
  (NOT five): the FIRST is the chat surface itself, used to return
  to agentView from a module view.

    ⌂   个人 Agent  (agentView)  →  the ImNav + AgentFlow surface
                                     (returns from any module view)
    ▣   项目管理   (projectView) →  module view, projectCards data
    ✓   任务视图   (taskView)    →  module view, taskCards data
    ◇   知识库     (knowledgeView)→  module view, knowledgeCards data
                                     (renders in 2-col `.two` grid)
    ◎   组织管理   (orgView)     →  module view, orgCards data
    ⌕   审计视图   (auditView)   →  module view, auditCards data

    [separator]
    ▦   工具栏 toggle (toggles toolsOpen, persists in localStorage)
    ?   帮助       (deferred, can route to /docs or be inert)

  Selecting any Rail item OTHER than ⌂ (agentView) sets
  className="moduleMode" on .app, which (legacy CSS lines 95-97):
    • hides .im (ImNav)
    • shrinks layout to: var(--rail) minmax(0,1fr) 6px var(--tools)
    • the module view's content (cards) renders in .main

  Returning to ⌂ removes moduleMode and restores ImNav + AgentFlow.

  ProjectBar stays visible in module mode (the user is still scoped to
  a project). Per Q-B answer (§10), the Cell pill is opt-in: detail
  pages show all items the user's license/auth permits, with project
  filter as one optional toggle, not a hard scope.

--------------------------------------------------------------
  5. PROJECTBAR — SCOPE HINT, NOT NAVIGATION
--------------------------------------------------------------

ProjectBar (34px) is global, lives between TopBar and the layout grid.
It surfaces:

    [●] 当前项目: <project name>                       ← left crumb (read-only)
    [当前工作台: <derived from active stream>]        ← optional sub-crumb
    [Personal*] [Cell] [Department] [Enterprise]       ← scope tier pills
                  (* = default-on; others opt-in)

The pills are the user-facing projection of LicenseContextService scope
decisions (per 04-28 memo — Personal / Cell / Department / Enterprise
tiers). PROJECT (= Cell tier) IS AN OPTION SCOPE, NOT DEFAULT. Default
in 我的 Agent stream: Personal ON; Cell/Department/Enterprise OFF.
User toggles Cell on to broaden context to the active project per turn /
per stream. (Q-B answer 2026-05-01.) This is a correction from earlier
drafts that assumed Personal+Cell were both default-on.

Project switching happens via TopBar 切换项目 dropdown, NOT by clicking
the crumb. The crumb is read-only context.

  9. ACCEPTANCE CRITERIA (HOW WE KNOW THE SHELL IS DONE)
--------------------------------------------------------------

A user with 3 projects, 12 groups across them, 4 DMs sees:

  1. On login, lands in their personal stream of the most recent
     project (no dashboard, no project list page as primary).
  2. ImNav shows: 我的Agent (1 entry) · 群组 (12 flat) · 私聊 (4).
     No "项目" section. No specialist-agent picker.
  3. ProjectBar shows the active project's name + tier pills. Clicking
     it does nothing (read-only crumb).
  4. TopBar 切换项目 dropdown lists the 3 projects; selecting one
     swaps the personal stream + ProjectBar context, ImNav unchanged.
  5. Clicking a group in 群组 — whether it's a project main stream or
     a sub-room — opens it in Main. The user CANNOT tell from the UX
     which underlying kind it is.
  6. Crystallizing a decision in any group: the vote tally line reads
     "Voting with this group's N members" where N is the group's
     member count, regardless of underlying kind. (BE already correct
     per §3; FE just needs uniform copy.)
  7. Rail 📋/📌/📚/🏛/🔍 swaps Main to the corresponding /detail/* page
     and hides ImNav. Workbench stays. Returning to chat (Rail logo W
     or first icon) restores ImNav.
  8. Workbench ▦ toggle hides/shows Workbench column; preference
     persists.
  9. Splitters between ImNav|Main and Main|Workbench drag-resize.
 10. ⛶ in chat header enters immersive mode (only TopBar + Main
     remain visible).

If all 10 hold, the shell is prototype-faithful and the user's
05-01 frustration is resolved.

  Q-B. ProjectBar — project as OPTION scope, not default.
       USER ANSWER (2026-05-01): "project also should be a option
       scope, not default. detail shows stuffs accoding to the user's
       auth or license allow them to see. so still project is an
       option to adjust scopes."

       APPLIES: the four scope-tier pills [Personal] [Cell] [Department]
       [Enterprise] are toggleable. Default state in 我的 Agent stream
       is Personal ON, Cell/Department/Enterprise OFF. User can OPT IN
       to Cell (current project) for context broadening per turn / per
       stream. Detail pages (Rail icons) filter by user's
       license/auth set; project is one OPTIONAL scope filter on top.

       Implication for ProjectBar mount: it's still global (top of
       shell), still shows the active project as a crumb (so the user
       knows which Cell scope WOULD be applied if they toggle it on),
       but the Cell pill is OFF by default. This is a behavior change
       from the prior assumption that Personal+Cell were both default-
       on; previous draft of this spec is corrected here.

  Q-C. (Folded into Q-B answer.) Rail detail pages scope by user's
       license/auth set; project is one optional filter, not a
       hard scope. Implementation note: when a user enters a Rail
       detail page (e.g. 知识库), the page renders ALL items their
       license tier permits, with a project filter chip the user can
       toggle on to narrow to the current project.

  Q-D. ImNav scaling for ~100 groups.
       USER ANSWER (2026-05-01): "by recency."

       APPLIES: 群组 section sorts by stream.last_activity_at desc.
       No search input as a primary affordance. If list grows beyond

       Section header collapse remains available for users who want

  Q-E. Group default name.
       USER ANSWER (2026-05-01): "maybe default use the project name
       the creater of the group is in."

       APPLIES: when StreamRow.name is NULL and the stream needs a
       display label in ImNav / DecisionCard / etc., resolve as:

         1. StreamRow.name if set (ad-hoc rooms named at creation)
         2. Else, ProjectRow.name of the project the creator of the
            stream belongs to. For kind='project' main streams that's
            unambiguous (project_id is set on the stream itself); for
            kind='room' streams whose project anchor is set, it's also
            project_id; for kind='dm' streams it's the partner's name.
         3. Else, fallback to "未命名群组" or stream id suffix.

       Server-side: GET /api/streams (or whatever the new ImNav hook
       calls) should resolve display_name in this order and return it
       on each row, so the FE doesn't have to join.
