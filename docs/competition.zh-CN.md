# WorkGraph AI — 参赛作品说明

**面向**: 字节跳动协作赛道评审
**作品名**: WorkGraph · Coordination as a graph, not a document
**状态**: 已上线可演示 · https://graphflow.flyflow.love
**关联材料**: 95 秒演示视频脚本 `workgraph-mock-data-v2/docs/video_demo_script.zh-CN.md`

---

## 一、一句话说清楚

**WorkGraph 是群体的共享操作系统,外化为一张知识图谱。** 每个成员登录后看到的不是共享聊天室或文档树,而是自己与本人智能体的私密对话。AI 承担所有的上下文整理、路由、决策结晶工作,人只做判断。

---

## 二、核心洞察:为什么不是 Notion+AI、不是 Lark+AI

过去十年,协作工具的演化停留在 Feishu / Lark / Notion / Slack 这条线上。它们的共同前提是:**信息的单位是文档**。因为在 AI 到来之前,只有文档能把多个人的思维冻结成共享的快照。文档是 pre-AI 时代最优秀的协调原语。

但 AI 改变了底层:协调层本身开始能思考、路由、记忆、结晶。此时若仍把文档当作信息单位,就是用上一代工具装上机器引擎 —— 还是飞机形态,只是多了油门。

**真正的后 AI 协作,信息单位应该变成"轮次"(turn)。** 每一次人与 AI 的对话、一次转发、一次决策,都是图上的一个节点或一条边。文档是这些轮次累积后的自动渲染产物,不是输入。

这是 Ford 在物理世界做过的事。Ford 没有发明汽车,发明的是把工人摆进工作流、让工作流成为主体的组装线。知识工作业至今仍在 "工匠时代":每个人按自身技能自定义工作方式,协作靠共享文档串联。**WorkGraph 是第一台认识论意义上的装配线 —— 工作流是主体,人是其中承载着判断的节点。**

与 Ford 的关键不同:物理装配线因为工人判断不重要而去人性化;认识论装配线的全部价值就在节点的判断上。机器负责上下文保全,人负责抉择。

## 三、目标用户与场景

**首批目标**: 10–30 人的知识密集型小团队 —— 独立游戏工作室、早期创业公司、创意机构、研究小组。这些团队的特征:

- 跨学科协作密集 (设计 × 工程 × 产品 × QA × 运营)
- 每天有多次"需要对齐"但不值得开会的事
- 文档很多但没人读,会议很多但没记住
- 已经用 Feishu / Notion / Slack 之一,仍觉得协作成本高

**不适合**:执行类重复性团队 (工厂调度、呼叫中心);纯个人作业无协作;超大型组织 (500+ 人,需要子图切片,属于 v2 再讨论)。

**示例场景 Moonshot Studios** (演示所用的虚构独立游戏工作室):6 人团队,距发售 4 周,面对 Boss 难度 / Switch 性能 / 永久死亡机制三个核心决策。演示视频见 `video_demo_script.zh-CN.md`。

## 四、核心原语

**1. 每位成员一个全局子智能体 (sub-agent)**

每个用户拥有唯一且跨项目的智能体身份。用户进入某个项目时,智能体载入该项目的图状态 + 知识库切片 + 用户自己的最近输出。这匹配 `vision.md §4.3` 的响应画像 (response profile) —— 成员是他们"生产关系的承担者",他们的智能体也继承这一属性。

**2. 流是唯一界面**

系统中所有界面都是一个"流"。流有三种类型:

| 类型 | 用途 | LLM 活跃度 |
|---|---|---|
| 个人项目流 ★ 主界面 | 用户与本人智能体的私密对话,项目级 | 主动回答、澄清、提议路由 |
| 团队公共流 | 少数群体同步时刻用,默认极少进入 | 被动 |
| 1:1 私信流 | 紧急呼叫 + AI 路由的双向记录层 | 被动 |

团队公共流存在但不鼓励默认使用 —— 绝大多数协调经路由发生在个人流之间。

**3. 路由 (Routing) 取代会议**

当 Maya 在个人流输入 "永久死亡改只作用于 Boss,工程代价多大?",她的智能体识别这属于工程权威领域,提议 `路由给 Aiko`。Maya 点击 "询问 Aiko",父智能体把信号发给 Aiko 的子智能体。Aiko 不看 Maya 原文,看到的是带有结构化选项的 `routed-inbound` 卡片:

- **标签** (Label):候选行动
- **背景** (Background):来自图谱的上下文
- **理由** (Reason):为什么这是候选
- **交换代价** (Bargain):采纳的代价
- **权重** (Weight):智能体的建议强度 0–1

Aiko 一键 **Pick this**,或自定义回复。回复经父智能体合成为 "Aiko 反提纪念品复活方案,保留永久死亡,2 周工程投入",进入 Maya 的个人流。Maya 点 **接受**,⚡ 决策结晶为图节点,血缘可追。全过程 90 秒,零会议、零文档书写。

**4. 决策结晶 (Decision Crystallization)**

每个被接受的抉择成为图上的永久节点,包含:

- 发起者 & 接受者
- 源信号 (原始问题)
- 候选选项与各自权重
- 最终选择 + 理由
- 被影响的任务 / 风险 / 里程碑

3 个月后任何成员问 "我们当时为什么选纪念品复活?",智能体从图谱直接答出因果链,不依赖任何人的记忆。这就是 `vision.md §5.10` 的 "why chain"。

**5. 图谱即状态,而非展示**

系统里所有的"状态"都是图谱的投影:

- 任务表 = 图的任务节点投到二维表
- 风险面板 = 图的风险节点按严重度排序
- 仪表盘 = 成员 × 任务 × 风险的摘要切片
- 复盘文档 = 决策血缘的自然语言渲染

**人从不编辑这张图,人只是输入轮次。图自己生长。**

## 五、产品界面 (v1 已上线)

**主视图 `/projects/[id]`**
个人项目流。左侧全局侧边栏:首页 / 项目 (可展开为 "我的会话 / 团队会议室 / 状态 / 知识库 / 渲染文档 / 详细视图") / 私信。

**主界面交互特点**

- 输入框上方的**预演卡**:你打字 1 秒后,智能体预读草稿,提示"这看起来是路由到 Aiko 的请求"、"背后会调用 kb_search 技能"等。用户提交前就能修正。
- 工具调用可见化:智能体使用 `kb_search` / `recent_decisions` / `risk_scan` / `member_profile` 等技能时,工具调用 + 结果都作为折叠卡片出现在流里。用户知道智能体在干什么。
- 路由提议卡 + 一键 "询问 X" 按钮,决策结晶卡 ⚡ 带可展开血缘,漂移提示卡在愿景与实际产出背离时主动浮出。

**附属能力**

- 多语言切换 (中 / 英),UI 按用户偏好渲染;知识库按语言自动过滤。
- 自动生成复盘文档、交接文档 —— 点击按钮,智能体从图谱生成含决策血缘引用的 Markdown 报告。
- 演示数据覆盖 EN + ZH 双项目,十条任务、四条里程碑、四条风险、三条决策、二十个维基页面,保证演示可重现。

## 五B、页面详解(配图位)

本节为每一个主要页面提供一段"评审导览"。每段先用一句话说明"这个页面为什么存在",再用 2–3 句说明屏幕上实际在发生什么。图片由选手现场补位。

### 1. 登录 / 首页 `/`

**为什么存在:** WorkGraph 不强调登录页的仪式感 —— 页面左侧是一个不断生长的图谱演示,右侧是极简登录框。未登录访客直接看到"产品的本体就是一张图",登录后首页变成"你当前要处理什么"的个人汇总。

**屏幕上在发生什么:** 登录后,首页分为四块 —— 待处理信号(Edge 抛给你的提议)、在做的任务(按项目聚合)、项目列表(未读徽章实时闪烁)、私信与审批。这四块合起来是"你个人今天的工作指令",不是公司级公告墙。

**配图建议:** 登录态 + 未登录态各一张,重点展示登录后"四块个人汇总"的紧凑排布。

---

### 2. 项目列表 `/projects`

**为什么存在:** 用户通常同时属于 2–4 个项目,本页是跨项目跳转的最短路径。每张卡片上显示项目名、最近一条来自 Edge 的提议、未读徽章。

**屏幕上在发生什么:** 点击 "+ 新建项目"立即触发需求智能体(RequirementAgent)走入场流程 —— 智能体会以对话形式收集一句话目标、成员与角色、关键约束,而不是丢一张空白表单给用户。

**配图建议:** 一张覆盖 2–3 个项目卡片的网格视图;如果能顺手把 "+ 新建项目"对话框叠上去更佳。

---

### 3. 个人项目流 `/projects/[id]` —— 主战场

**为什么存在:** 这是全产品最核心的页面,也是评审最应该停留的画面。用户进入项目后看到的不是共享群聊,不是文档树,而是一条只属于自己与本人子代理的私密对话流。他个人的注意力不会被团队杂音撕碎。

**屏幕上在发生什么:**
- **预演卡(Rehearsal):** 输入框上方 1 秒实时刷新。用户还没按发送,智能体已经对草稿作出四种分类之一 —— 回答 / 追问 / 路由建议 / 静默。发送前可修正措辞,相当于"发前彩排"。
- **工具调用可见:** 智能体触发 `kb_search` / `recent_decisions` / `risk_scan` 等技能时,调用入参与结果以折叠卡片形式嵌入流中。不是黑箱,是可展开的侧证链。
- **路由提议卡:** 当智能体判断"这件事你不用自己答,可以路给 Aiko",会出现一张带 1–3 个"询问 X"按钮的卡片。每个按钮旁还有**预读按钮(Stage 2 新增)** —— 点一下,目标成员的子代理会基于他的角色技能 + 个人技能,先拟一份答复给你看,你满意就直接关掉不打扰本人。
- **决策结晶卡 ⚡:** 一旦多轮对话收敛出一个决定,智能体自动把它抽成独立节点,带可展开血缘,未来任何引用都指回这一节点。
- **漂移提示卡:** 如果项目愿景与当日产出明显背离,DriftAgent 主动在流中浮出提醒,这不是被动等用户来查,而是"会主动敲门"。

**配图建议:** 至少两张 —— 一张完整的流视图(预演卡 + 工具调用 + 决策结晶全景),一张路由提议卡配预读面板的特写。

---

### 4. 技能图谱 `/projects/[id]/skills` —— 群体的资本

**为什么存在:** 知识库(KB)记录了小组"知道什么",技能图谱记录小组"能做什么"。每位成员绑定两类技能:**角色技能**(由所任角色派生,随岗位传承)+ **个人技能**(入职时自述 + 日常工作被图谱观察验证)。这两类合起来,是小组除知识库之外的第二类资本。

**屏幕上在发生什么:**
- **项目主视图:** 卡片网格里列出每位成员的两类技能,绿色徽章代表"自述 × 观察双重验证",琥珀色徽章代表"尚未被实际工作验证"。
- **小组聚合(仅项目主可见):** 一个 4 行摘要 —— 角色技能覆盖、声明能力合并、观察到的技能合并、**未验证声明**。最后一行是"谁说自己会,但还没被观察到",是小组能力识别的盲区扫描。
- **受限视图(普通成员):** 成员只看得见自己那张卡,其他人卡片与小组聚合不对普通成员开放,规避"公开技能评分"带来的心理负担。

**配图建议:** 一张项目主的满视图,一张普通成员的受限视图对比。

---

### 5. 团队会议室 `/projects/[id]/team`

**为什么存在:** 保留一个共享流面向少数必须多人同屏的瞬间(晨会、同步会)。设计上并不鼓励高频使用 —— 默认路径是让智能体把信息路由到相关人的个人流,而不是全员喊话。

**屏幕上在发生什么:** 与个人流 UI 一致,但标题明确"团队会议室"。消息在此广播,但仍然过 Edge 翻译 —— 每位成员的子代理会把相关消息再二次整理进自己的个人流。

**配图建议:** 一张团队流视图,配一条广播消息被分流到两个不同成员个人流的对比图更有说服力。

---

### 6. 状态面板 `/projects/[id]/status`

**为什么存在:** 当用户是"审计角色"(任务范围成员 / 财务 / 顾问等非核心贡献者),他不应该进个人流看冗长对话。状态面板是他的首屏:只读、聚合、按区块。

**屏幕上在发生什么:** 五个区块 —— 成员、任务摘要、风险、决策、工件。顶部还有"一键渲染"按钮,点击后调用 RenderAgent 生成最新的复盘或交接文档并缓存。适合给需要"快速对齐"的外部审阅者。

**配图建议:** 一张完整面板截图,若能把"一键渲染"生成的 Markdown 文档叠加更好。

---

### 7. 知识库 `/projects/[id]/kb` 与条目详情 `/projects/[id]/kb/[itemId]`

**为什么存在:** 小组共享的"外脑"。与 Notion 的不同在于:知识库的每条条目都可以被智能体的 `kb_search` 技能引用,引用会在个人流里以来源链接的形式出现,而非被抄袭进回答中。

**屏幕上在发生什么:** 列表页支持按语言过滤 + 搜索;详情页是一个简洁的维基页面 + 反向链接(本条目被哪些决策 / 任务引用过)。双向链接让知识库与项目事件互相支撑。

**配图建议:** 列表页一张、条目详情页一张。

---

### 8. 渲染工件 `/projects/[id]/renders/[slug]`

**为什么存在:** 把"项目历史"以 Markdown 形式晒出来的一键入口。`postmortem` 为项目复盘,`handoff:<user_id>` 为某位成员的交接文档。这些不是 LLM 凭空写,而是从决策 / 风险 / 任务图谱严格 grounding 出的结构化产出,引用都是可点击回源的节点链接。

**屏幕上在发生什么:** 左侧是工件全文,右侧是"引用到的节点"列表,点击跳转到对应的决策 / 任务 / 风险节点详情页 —— 所以交接文档不是一堆字,而是可追根的记录。

**配图建议:** 建议使用复盘文档为例,展示"引用→节点→血缘"三跳。

---

### 9. 图谱可视化 `/projects/[id]/detail/graph`

**为什么存在:** 前面所有页面都是从特定视角"切片"看图谱,本页是"图本身"。任务、决策、风险、里程碑、目标、冲突等节点以颜色区分,依赖与引用以边相连,支持缩放、拖拽、聚焦。

**屏幕上在发生什么:** 用户可以点选任意节点打开详情页,也可以追溯"某个决策为什么长这样"的血缘链。图视图是证据链而不是装饰,尤其在评审"这个产品的记忆到底记了什么"时最直接。

**配图建议:** 一张中等密度的图谱全景(15–20 节点),重要决策节点高亮。

---

### 10. 详情视图(表格形态) `/projects/[id]/detail/{tasks,risks,decisions,conflicts,plan,events}`

**为什么存在:** 图谱适合讲故事,但有些场景需要表格的稠密性 —— 评审或外部审计时,"给我一份所有任务/风险清单"的请求比"给我一张图"更常见。本组页面是图谱的另一种投影。

**屏幕上在发生什么:**
- **tasks:** 飞书风格稠密表格,每行显示状态、负责人、里程碑、交付物、估时、依赖。带颜色状态芯片,支持排序。
- **risks:** 按严重度着色排序。
- **decisions:** 审计形态 —— 每条决策带 ID、参与方、结果、理由。
- **conflicts:** 冲突摘要 + 冲突卡片(开放 / 严重 / 高 / 中 / 低),每条冲突含解决选项。
- **plan:** 规划任务 DAG,由 PlanningAgent 从 Clarification 结果衍生。
- **events:** 项目所有变更的时间线(任务更新、决策作出、风险标注……),实时刷新。

**配图建议:** 至少 `tasks` 与 `events` 各一张 —— 前者证明"我们也能做表",后者证明"所有变更都有审计"。

---

### 11. 节点详情页 `/projects/[id]/nodes/[nodeId]`

**为什么存在:** 任何引用都需要回源 —— 决策血缘、Why-Chain、工件引用点的链接,最终都指向本页。

**屏幕上在发生什么:** 页面先识别节点类型(任务 / 决策 / 风险 / 交付物 / 目标 / 里程碑 / 冲突),再按节点类型渲染对应字段与血缘链;Why-Chain 引用会连带展示"推理路径"上经过的所有节点。

**配图建议:** 建议用一个决策节点作为例,展示血缘链 + 相关节点面板。

---

### 12. 私信 `/streams/[id]`

**为什么存在:** 紧急情况下人与人仍需直接对话,WorkGraph 不禁用私信,但会在私信旁留一个"AI 路由记录层" —— 你与对方的子代理也会留下他们参与路由的痕迹。

**屏幕上在发生什么:** 一对一聊天 UI,消息同时显示在两位成员的个人流里(作为来源注明)。如果私信是由 AI 路由推送的,会保留路由提议卡的原始血缘。

**配图建议:** 一张对话视图 + 旁边展示对应个人流中出现的"路由记录层"。

---

### 13. 个人资料与观察 `/settings/profile`

**为什么存在:** 用户能看到"自己被图谱观察到做了什么"—— 近 30 天消息数、解决的决策数、拥有的风险数、路由应答数。这不是打分,是"我的子代理的学习素材",因为这些信号会反过来指导子代理如何回应其他人的路由请求。

**屏幕上在发生什么:** 一张用户卡,角色自述 + 声明能力 + 观察到的技能标签 + 近期活动节奏统计。用户可以主动修改自述与语言偏好。

**配图建议:** 一张个人资料页全景。

### 14. 管理台 `/console/[id]` (运营视角)

**为什么存在:** 管理员 / 交付视角需要看历史版本的渲染产物(版本化的复盘与交接),以及项目整体运行状态。与状态面板的差别在于它面向"谁在管项目",而不是"项目现在是什么样"。

**屏幕上在发生什么:** 项目状态摘要 + 已生成工件的历史版本列表(含时间戳与触发者)。典型用途:复盘会前,管理员一键把所有版本的复盘摆出来看变化。

**配图建议:** 一张管理台全景 + 工件历史列表侧栏展开。

## 六、技术架构

单节点部署,三种主要容器 + 一个智能体隧道:

```
Cloudflare Tunnel
     ↓
  Nginx
     ├─ / → Next.js 15 Web (React 19, next-intl 中英双语)
     ├─ /api/* → FastAPI (Python 3.11, SQLAlchemy + aiosqlite)
     └─ /ws/* → FastAPI WebSocket (CollabHub + Redis 消息总线)
                    ↓
              SQLite (workgraph.db)
                    ↑
              LLM: DeepSeek (OpenAI 协议兼容,provider-agnostic)
```

**为什么 DeepSeek**:国内可达、成本低、合规友好,且提示词合约是 provider-agnostic (随时切换到 Kimi / 通义等)。

**智能体层**

七个专职 LLM 智能体分工:

| 智能体 | 职责 |
|---|---|
| EdgeAgent | 每位成员的子智能体 —— 对话、澄清、工具调用、路由提议 |
| Clarification | 需求阶段的澄清问题生成 |
| Conflict Explanation | 执行冲突的解释与选项生成 |
| Planning | 从需求 v+1 生成任务 DAG + 里程碑 |
| DriftAgent | 周期扫描愿景 vs 实际产出的偏差 |
| MembraneAgent | 外部信号 (Git / Steam / RSS) 的分类与路由,带注入防御 |
| RenderAgent | 复盘 / 交接 Markdown 渲染,带决策 ID 引用落地 |

每个智能体的提示词存于版本化的 Markdown 文件,恢复梯度一致:JSON 模式 → 三次重试 → `manual_review` 兜底。观测体系:`agent_run_log` 表 + trace_id 贯穿,token 消耗、延迟、重试次数全记录。

**数据模型亮点**

- `StreamRow` 统一流原语,三种类型 (project / personal / dm)
- `RoutedSignalRow` 路由边 (source/target/framing/options/reply,带血缘)
- `DecisionRow` 决策结晶,可回溯到源信号 + 冲突 ID
- `MembraneSignalRow` 外部图谱入口,注入防御策略化

图谱即 DB。没有隐藏状态,没有 `current_stage` 之类的反范式字段。任何"当前阶段"都是对图的实时查询 (decision 1E)。

**测试体系**

- 后端 139 个测试覆盖 signal-chain / streams / routing / personal / skills / kb / drift / membrane / render / ws
- 前端 TypeScript 严格模式,所有代码 tsc 通过
- 每个智能体带 stub LLM 单元测试 + 恢复梯度覆盖

## 八、差异化

| 维度 | Feishu / Lark 文档+妙记 | Notion AI | Slack + 自建 GPT | **WorkGraph** |
|---|---|---|---|---|
| 协调单位 | 文档 | 文档 + 数据库 | 消息 | 轮次 + 图节点 |
| AI 角色 | 辅助编辑、摘要 | 问答、AI Blocks | 问答 bot | 每位成员的私人智能体 + 路由枢纽 |
| 决策留存 | 依赖会议纪要 / 文档更新 | 依赖手动记录 | 依赖 thread 找回 | 自动结晶为图节点,带血缘 |
| 跨成员协调 | @ 人 + 文档共享 | 文档评论 | 频道 @ 人 | 智能体路由,带结构化选项 |
| 外部信号 | 人工转发 | 人工粘贴 | 人工粘贴 | 膜结构自动分类入图,带注入防御 |
| 为什么我们不同 | 它以文档为主体,我们以轮次为主体 | 它是 AI 写作,我们是 AI 协调 | 它是助手,我们是枢纽 | **重新组装了协调基元** |

**对 Feishu 的态度:尊重 + 继承**
Feishu 把文档协同做到极致,定义了 pre-AI 时代的最优解。WorkGraph 不想也不能替代 Feishu 的所有场景 —— 全员通知、审批流、OA、企业邮箱仍然属于 Feishu 的舒适区。但在 "小团队日常决策" 这个最高频的场景里,我们提供了一个不同维度的解法:**让文档从输入退为产出,让聊天从时间线退为可追溯的图**。

## 九、商业与推广

**短期 (3 个月内)**:免费对外,吸 20 个 10 人以下种子团队 (独立游戏 / AI 创业 / 设计工作室),出 3 份产品使用白皮书。

**中期 (6–12 个月)**:对 Feishu / 钉钉的企业做"伴生部署" —— 保留它们作主力 OA,WorkGraph 做特定项目的协调层。定价:10 人以下免费,超额按月 ¥50/人。

**长期愿景**:开放图谱 API,允许第三方以"响应画像" 的形式接入专业智能体 (法务、财务、安全审计等),WorkGraph 成为中小团队的共享基础设施。

**与字节生态的协同想象**:

- 豆包 / 火山方舟 LLM 可直接替换 DeepSeek,provider-agnostic 架构无需重构。
- Feishu 的 Bot 协议可作为外部膜结构的接入点,把 WorkGraph 的决策结晶反向推送到 Feishu 文档或群聊里。
- 飞书多维表格 / Bitable 可作为 WorkGraph 图谱的外部镜像,面向喜欢表格视图的用户群体。
